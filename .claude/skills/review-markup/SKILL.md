# コンポーネント マークアップレビュー

TSX と対応する CSS Module を対象に、**HTML セマンティクス** と **CSS Modules ルール** の両面からレビューし、問題を修正する。

---

## 手順

1. 対象の TSX ファイルと対応する `.module.css` を読む
2. 下記のチェック観点に沿って問題を洗い出す
3. 問題を一覧で報告し、ユーザーと合意してから修正する

---

## チェック観点

### HTML セマンティクス

#### 要素の選択

- 見た目ではなく意味で要素を選ぶ。汎用コンテナ (`div`, `span`) は、適切なセマンティック要素がない場合のみ使う
- ランドマーク要素 (`main`, `nav`, `header`, `footer`, `section`, `article`) を適切に使っているか
- 見出しレベル (`h1`〜`h6`) が論理的な階層を反映しているか。見た目のサイズ調整で見出しレベルを選ばない
- リスト (`ul`, `ol`) は意味的に並列な項目群に使う。一文の中の改行位置をデザインしている場合は `<br />` が適切

#### 見出しの範囲

- 視覚的に一つの見出しとして機能するテキストが、`h1`〜`h6` の外に分断されていないか
- スタイリングのために見出し内の一部テキストを外に出している場合、`<span>` で `h1` 内に含めてスタイルを分離する

#### アクセシビリティ

- 装飾画像には `aria-hidden="true"` が付いているか
- 意味のある画像には `alt` テキストがあるか
- インタラクティブ要素にアクセシブルな名前があるか

### CSS Modules ルール

`css-modules` スキルのルールに従っているかチェックする。主要な観点:

1. **フラット構造**: ネストで詳細度を上げていないか（単一クラスセレクタのみ）
2. **クラス名**: 意味が明確か（`.a`, `.b`, `.c` のような命名は NG）
3. **レスポンシブ**: モバイルファースト (`min-width: 768px`) か
4. **CSS 変数**: マジックナンバーの代わりに `--space-*` 等を参照しているか

### クラス名の命名

- コンポーネント内での役割を反映した名前にする
- 1〜2語で、何をスタイリングしているか読み取れること
- 見た目ではなく構造・役割を表す名前にする（`.redText` ではなく `.error`）
