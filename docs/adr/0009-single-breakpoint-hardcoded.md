# 0009: ブレークポイントは固定セットをハードコードする

- ステータス: 採用
- 日付: 2026-06-12
- 改訂: 2026-06-12 — Works のカードグリッドで lg が必要になり、「768px 1本」から「768px / 1024px の固定セット」へ変更

## コンテキスト

これまで SP のみを対象にデザインしてきたが、レスポンシブ対応を始めるにあたりブレークポイントの管理方法を決める必要がある。

メディアクエリの条件には CSS 変数（`var()`）が使えないため、値を一元管理するには何らかのツールが要る。検討した選択肢:

1. **ハードコード + 規約**: 素の CSS のまま。range 構文 `@media (width >= 768px)` で書く
2. **PostCSS custom-media**: `@custom-media --md (width >= 768px)` で名前付き管理。Vite が postcss.config を自動で拾うため CSS Modules の配線（ADR 0003）には触れない
3. **Lightning CSS**: `css.transformer` の差し替えで `@custom-media` がネイティブに使える。ただし CSS Modules の処理系ごと置き換わるため、3点セット配線への回帰リスクがある
4. **Sass**: 変数 + mixin。ブレークポイント1個のためにプリプロセッサを足すのは過剰

## 決定

ブレークポイントは **Tailwind 相当の固定セット2本**とし、各 module に直接ハードコードする。ツールは導入しない。

| 名前 | 値 | 用途 |
|---|---|---|
| md | `@media (width >= 768px)` | SP / それ以外の基本分岐 |
| lg | `@media (width >= 1024px)` | グリッド列数など広い画面での追加分岐 |

- range 構文（`width >=`）で統一し、`min-width` / `max-width` 表記は使わない
- モバイルファースト（ベース = SP、上記値以上で上書き）
- 基準値は `app/styles/globals.css` 先頭のコメントを単一の真実とする
- **この2本以外の値を勝手に増やさない**

## 根拠

- 固定2本なら「一元管理」の価値はまだ薄く、ツールの導入・維持コストに見合わない
- 表記を `width >= 768px` / `width >= 1024px` の2パターンに固定することで、変更時も grep 一発で全箇所を更新できる
- CSS Modules の繊細な配線（ADR 0003 / AGENTS.md の3点セット）に一切手を入れずに済む

## 結果・トレードオフ

- 値の変更は全ファイル置換になる（表記統一が前提条件。レビューで `min-width` 混入を弾くこと）
- **3本目が必要になった、または値の組が頻繁に変わるようになった時点で PostCSS custom-media へ移行する**。Lightning CSS は配線の検証コストに見合うリターンが出るまで採用しない
